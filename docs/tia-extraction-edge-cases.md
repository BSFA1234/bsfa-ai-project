# TIA 블록 추출 — 엣지/에러 케이스 정리

> **독립 문서.** 추출 중 만나는 예외 상황과 그 처리법을 누적 기록한다. 새 케이스가 나오면 여기에 추가.
> 관련: 방법론 `docs/tia-openness-extraction.md`, 스킬 `.claude/skills/tia-block-extract/`.

## EC-1. `IsConsistent=False` = "에러"가 아니라 "아직 컴파일 안 함" (2026-06-20)

### 증상
- 블록 추출 시 일부 블록이 `IsConsistent=False`로 나와 `block.Export()`가 거부됨 → 추출 스킬이 `SKIP`.
- (실사례) `02_제어/03_OP`의 OP01·02·03·06·07·08이 미컴파일 상태라 추출에서 빠짐 (OP04·05·09·10만 추출됨).

### 정확한 의미
- **소스(사람이 그린 LAD) ≠ 컴파일된 것(PLC 실행형 + 참조 검사 완료).**
- 블록을 **편집하면** TIA는 "이 최신본은 아직 번역·검사 안 함"으로 표시 = `IsConsistent=False`. **이것은 에러 여부와 무관한 기본 상태**다(편집 후 컴파일 안 하고 저장하면 이 상태로 남음).
- Openness `Export`는 **일관(컴파일된) 블록만** 내보낸다 → 그래서 미컴파일 블록은 추출이 막힌다.

### 판별법 (핵심)
미컴파일 블록은 **시험 컴파일 한 번**으로 두 경우가 깔끔히 갈린다:
- `Compile()` 결과 **errors=0 → 그냥 컴파일만 안 돼 있던 것**(고칠 곳 없음). 컴파일 후 `IsConsistent=True` → **바로 추출 가능**.
- `Compile()` 결과 **errors>0 → 진짜 에러**. 메시지의 `Path`(위치) + `Description`(내용)으로 어디가 왜 틀렸는지 알 수 있음 → **사람이 수정 필요**.

### 처리 정책 (스킬에 반영됨)
- 추출 스킬은 미컴파일 블록을 만나면 **자동 컴파일(`-AutoCompile` 기본 ON)** 후 추출한다.
- 컴파일해도 에러가 남는 블록은 추출하지 않고, **에러 위치+내용을 `OutDir\_extract_errors.txt`** 에 사람이 보기 쉽게 적는다.

### 실사례 결과 (CP_A4 / 02_제어/03_OP)
- OP01·02·03·06·07·08 시험 컴파일 → **전부 errors=0** (단순 미컴파일). 자동 컴파일 후 OP01~10 **10개 전부 추출 성공**.
- 즉 "사람이 고쳐야 한다"가 아니라 "컴파일만 하면 됐다"였음.

### 교훈 (왜 처음에 놓쳤나)
- `IsConsistent=False`를 "사람이 고쳐야 함"으로 **뭉뚱그려 가정**하고, 시험 컴파일(확인 비용 작음)을 안 했던 게 원인.
- 올바른 흐름: **미컴파일 발견 → 시험 컴파일로 (단순 미컴파일 / 실제 에러) 판별 → 보고** → 스킬의 AutoCompile 정책으로 자동화됨.

## EC-2. 경고: "Inputs or outputs are used that do not exist in the configured hardware" (2026-06-20)

### 증상
- 위 블록들 컴파일 시 **errors=0, warnings=1**, 경고 내용 = *"설정된 하드웨어에 없는 입력/출력을 사용 중."*

### 의미 / 처리
- **경고(Warning)지 에러가 아니다.** 컴파일·추출을 **막지 않는다**(추출 정상).
- 뜻: 블록이 참조하는 일부 I/O 주소가 현재 하드웨어 구성에 없음(예: 일부 모듈 미구성, 또는 해당 사본의 HW 상태). 엔지니어가 참고할 정보지 블록 자체 오류는 아님.
- 정책: 경고는 추출을 막지 않으므로 그대로 추출하되, 필요 시 컴파일 로그로 확인.
